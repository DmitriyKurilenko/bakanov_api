# Versioning & Commit Protocol

## Версионирование

Используется Semantic Versioning: `MAJOR.MINOR.PATCH`

- **MAJOR** — ломающие изменения API/поведения
- **MINOR** — новая функциональность, обратно совместимая
- **PATCH** — багфиксы, рефакторинг без изменения поведения

Версия хранится в файле `VERSION` (создаётся при первом релизе).

## Формат сообщений коммитов

```
<тип>(<область>): краткое описание

Тело (опционально): что и почему изменено.
```

### Типы
- `feat` — новая функциональность
- `fix` — исправление бага
- `refactor` — рефакторинг без изменения поведения
- `docs` — только документация
- `style` — форматирование, PEP 8 (без изменения логики)
- `test` — добавление/изменение тестов
- `chore` — конфигурация, зависимости, CI/CD
- `perf` — оптимизация производительности

### Области (scope)
- `crm`, `dashboard`, `integrations`, `users` — по приложениям
- `config` — settings, urls, celery
- `templates` — шаблоны
- `docker` — Docker, compose
- `deps` — зависимости (requirements.txt)

### Примеры
```
0.2.2-dev feat(crm): добавить генерацию доп. договора
fix(integrations): исправить парсинг webhook Novofon
refactor(dashboard): вынести логику отчётов в сервисы
style(crm): привести models.py к PEP 8
docs: создать структуру docs/
chore(docker): обновить Python до 3.13
```

## CHANGELOG.md

Создаётся при подготовке коммита. Формат:

```markdown
## [X.Y.Z] — YYYY-MM-DD

### Added
- ...

### Changed
- ...

### Fixed
- ...
```

## Правила для агентов

1. **НИКОГДА** не запускать `git commit` или `git add` самостоятельно.
2. «Подготовь коммит» = обновить VERSION, CHANGELOG.md, docs → СТОП. Пользователь коммитит сам.
3. Не трогать VERSION/CHANGELOG без явной просьбы «подготовь коммит».
